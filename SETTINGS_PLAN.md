# Раздел Settings — план реализации

План для раздела **Settings** (сейчас заглушка). Привязан к фактическому коду; помечено, что
готово «из коробки», а что требует бэкенда. Связанные документы: [`ARCHITECTURE.md`](ARCHITECTURE.md),
[`ROADMAP.md`](ROADMAP.md).

---

## 1. Где хранить настройки

| Категория | Хранилище | Комментарий |
|-----------|-----------|-------------|
| Персональные (тема, шрифты, цвета, личные дефолты) | `app.storage.user` (уже хранит `theme`) и/или `users.json` (метаданные) | без новых таблиц |
| Глобальные/админские (лимиты агента и т.п.) | **новая таблица `settings`** (key/value) | сейчас хранилища нет |
| Журнал AI (токены по пользователям) | **новая таблица `ai_log`** | сейчас только syslog/stdout (`event: llm_request`) |

Роли: админ = `fullmaster` (можно ввести узкие `users_admin`/`settings_admin`). Гейтинг доступа — по ролям,
как в Secrets/Objects (проверка `_role_allowed`/наличие роли).

## 2. Структура Settings (под-вкладки)

### 2.1 Внешний вид (персонально)
| Настройка | Как реализуется | Сложность |
|-----------|------------------|-----------|
| Тема (тёмная/светлая) | перенести существующий тумблер из шапки; `update_theme()` уже на CSS-переменных | тривиально |
| Цвета интерфейса | пользовательские оверрайды палитры (`THEMES` сейчас захардкожен) → `app.storage.user`, применять через те же CSS-vars | средне |
| Шрифт интерфейса | сейчас Orbitron/Roboto в CSS → выбор из списка, подстановка в CSS-var | средне |
| Тема/шрифт таблиц и кода | `codemirror_theme`, `aggrid_theme` уже в `current_state` (monokai / ag-theme-balham-dark) — вынести в выбор | средне |
| Личные дефолты | размер шрифта таблиц, лимит строк `PRINT`, `dpi` для `SHOW matplotlib`, тема CodeMirror | мелко |

### 2.2 Учётные записи
- **Админ (`fullmaster`)**: список УЗ; создать; **блокировать/разблокировать** (`users.enabled`);
  менять **роли** (`users.roles`); **сбросить пароль** (bcrypt; правило `REGEX_PASSWORD_RULE` уже есть);
  редактировать метаданные (`users.json`, включая `notify`-конфиг для команды NOTIFY).
- **Пользователь (не админ)**: сменить **свой** пароль (с проверкой старого), редактировать **свои**
  метаданные. Чужих УЗ не видит.
- Бэкенд (`db.py`): `list_users`, `create_user`, `update_user`, `set_user_password`, `set_user_enabled` + bcrypt.
- Дополнительно: управление **`access_networks`** (IP-whitelist уже в БД, но без UI) — админская настройка доступа.

### 2.3 AI-агент (админ) + журнал
- **Лимиты** (сейчас захардкожены) → таблица `settings`: число действий за сессию (`max_iterations`, сейчас 10),
  дефолтный таймаут запроса, дефолтная LLM, бюджет токенов на запрос/сессию.
- **Журнал агента**: таблица `ai_log` (timestamp, user, llm_model, provider, prompt_tokens, completion_tokens,
  total_tokens, duration_ms, ok) — заполняется из уже существующего `_log_llm_request`.
- **Детализация в UI**: сводка по пользователям (суммарные токены/запросы/время), фильтры по
  пользователю/модели/периоду (как в History).
- Дополнительно: дневной/сессионный лимит токенов на пользователя с мягким стопом; топ-потребители.

### 2.4 Система/безопасность (на будущее)
- Вынос **master key** из `front.py` (security-долг) + индикатор источника ключа (env/secret-manager).
- Подготовка к **API/MCP** наружу (токены доступа) — синхронно с Треком 2 в `ROADMAP.md`.

## 3. Оценка трудоёмкости
- **Дёшево/средне**: «Внешний вид» — персонально, без схемы БД (кроме мелких полей в `users.json`/`app.storage.user`).
- **Дорого/важно**: «Учётные записи» (CRUD + bcrypt + роли + блокировка) и «AI-журнал» (таблицы `settings`,
  `ai_log` + агрегация). Security-чувствительно — гейтинг по ролям обязателен.

## 4. Этапы (каждый — с ручным тестом владельца)
1. ✅ **Внешний вид** (реализовано): тема перенесена в Settings; выбор шрифта и размера для интерфейса
   и для таблиц/текстовых блоков; живой предпросмотр; «Сохранить»/«Сбросить»; persist в `settings`
   (scope `user:<username>`), применение через CSS-переменные (`--app-font`, `--app-font-size`,
   `--app-table-font`, `--app-table-font-size`) + override темы AG Grid. _Тема CodeMirror — отложена._
2. ✅ **Цвета интерфейса** (реализовано): пикеры палитры (Фон/Текст/Акцент/Карточки/Заголовки/Панели)
   поверх CSS-переменных, **отдельно для тёмной и светлой темы**; живой предпросмотр; persist в `appearance.colors`.
   Карточки/панели разделов (`.q-card`/`.uh-panel`), AG Grid (через `--ag-*`-переменные) и режим Quasar dark
   (`body--dark`) следуют теме/цветам; **тема редактора CodeMirror** выбирается из списка и применяется живо.
3. ✅ **Учётные записи — self-service** (реализовано): в Settings → «Учётная запись» — смена своего пароля
   (проверка текущего через bcrypt, правило `REGEX_PASSWORD_RULE`, повтор) и редактирование своих
   метаданных (`users.json`, JSON-редактор). Бэкенд: `set_user_password`, `update_user_metadata` (параметризованно).
   **Принудительный логаут**: session-epoch на пользователя в `settings(scope='session_epoch')`; смена пароля
   и блокировка (`set_user_password`/`set_user_enabled`) бампят epoch → `ui.timer` (15с) сверяет epoch+`enabled`
   и разлогинивает все сессии этого пользователя (работает и при действиях админа из этапа 4).
4. ✅ **Учётные записи — админ** (реализовано, роли `fullmaster`/`useradmin`): **AG Grid** со списком
   пользователей и фильтрами по колонкам (username/роли/метаданные); выбор строки → панель действий:
   блокировка/разблокировка (`set_user_enabled`), роли (`set_user_roles`), сброс пароля
   (`set_user_password`), **редактирование метаданных** (`update_user_metadata`), создание пользователя.
   Сброс/блокировка отзывают сессии пользователя (epoch). Запрет блокировать собственную УЗ. Смена своего
   пароля — с предупреждением и подтверждением (разлогин).
   ✅ **Разрешённые сети** (`access_networks`, роли `fullmaster`/`netadmin`): AG Grid с фильтрами,
   добавление (валидация CIDR + комментарий) и удаление; `create_access_network`/`delete_access_network`.
5. ✅ **AI-настройки + журнал** (реализовано, роли `fullmaster`/`aiadmin`): лимит «макс. действий агента
   за сессию» в `settings(scope='global', agent_max_iterations)` (читается в ReAct-цикле draw_ai);
   таблица `ai_log` заполняется из `_log_llm_request`; в Settings — сводка по пользователям (запросы/токены)
   и детализация запросов (AG Grid с фильтрами по пользователю/модели/времени).

Рекомендация: начать с этапа 1 (быстро, видимо, без риска), затем 3 (self-service пароль — базовая потребность),
затем админские (4) и AI-журнал (5).
