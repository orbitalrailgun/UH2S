# Парсер скриптов Harvester: как устроен, баги, предложения по исправлению

Документ описывает разбор скрипта DSL (`app/engine.py:command_parser` и связанные функции),
фиксирует найденные дефекты и предлагает исправления. Актуально на момент анализа.

---

## 1. Конвейер разбора

```
текст скрипта
  │  command_parser()                       app/engine.py
  ├─ 1. удаление комментариев  /* ... */
  ├─ 2. split по '|'           -> «сырые» команды
  ├─ 3. strip переводов строк/пробелов
  ├─ 4. выделение ключевого слова команды (первое слово) + остаток строки
  └─ 5. разбор каждой команды по её грамматике (match по ключевому слову)
        │
        ▼
   list[ {command, line, parsed, parsed_comment, ...поля по типу} ]
  │  commands_executor()                     engine.py
  ├─ DEF/CALC -> сбор variables
  ├─ process_injections() -> подстановка %(var)X в parameters
  ├─ резолв source/функции, проверка ролей и ТИПОВ параметров
  ├─ get_command_dependency() -> граф зависимостей
  └─ поэтапное выполнение -> result_map
```

Ключевой момент: **значения параметров на этапе парсинга остаются строками**; реальная
типизация происходит позже — в `commands_executor` через `get_variable_type` при сверке
с типами из `required` функции.

---

## 2. Грамматика команд

Команды разделяются символом `|`. Поддерживаются:

| Команда | Грамматика | Где разбирается |
|---------|-----------|-----------------|
| `DEF`   | `DEF <значение> AS <имя>` | регэксп `^(.+)\s+(as\|AS\|As\|aS)\s+(\S+)\s*$` + `get_variable_type` |
| `GET`   | `[APPLY:<data>(<col AS x>,...):[<unique>] ] <source>:<func>(<k=v>,...) AS <data>` | несколько регэкспов + split по `,` |
| `CALC`  | `CALC <a> <op> <b> AS <res>` (`+`,`-`,`*`) | регэксп; **исполнение не реализовано** |
| `SAVE`  | — | заглушка (`print("Internal Server Error")`) |
| `SHOW`  | `SHOW VARIABLES \| VAR x \| DATALIST \| TABLE d \| PLOT d x .. y ..` | заглушка |
| `NOTIFY`| `NOTIFY <notifier>("<текст>")` | регэксп `^\s*(\S+)\("(.*)"\)$` |

### Типизация значений (`get_variable_type`)
Строка распознаётся как: пустая → string; в кавычках → string; `True/true`,`False/false` → bool;
`^\d+$` → int; `^\d+\.\d*$` → float; `^\[.*\]$` + валидный JSON → list; `^\{.*\}$` + валидный JSON → dict.
Иначе — ошибка «unknow data type».

### Инъекция переменных (`process_injections`)
В значениях параметров `%(имя)X` заменяется значением переменной. Суффикс типа:
`s` строка, `i` int, `f` float, `b` bool, `l` список, `d` словарь, `x` «как есть» без кавычек.
Механика: `json.dumps(parameters)` → regex-замена → `json.loads` с валидацией.

### Зависимости (`get_command_dependency`)
Для `sqlite3_im`/`duckdb_im` имена таблиц извлекаются регэкспом `(FROM|JOIN)\s+([^\s;)]+)`
(с поправкой на CTE `WITH ... AS (`). Для `pandas_im`/`ollama` берётся `target_data`.
Для `APPLY` зависимость — имя входных данных.

## 3. Найденные дефекты

### P1 — КРИТИЧНО: параметры режутся по запятой без учёта вложенности
```python
parameters = re.split(r',\s*', command["raw_parameters"])   # GET
apply_columns = re.split(r',\s*', command["apply"]["raw_columns"])  # APPLY
```
Любое значение, содержащее запятую (JSON-список, SQL со списком колонок), обрывается
на первой запятой.

**Пример (реальный сбой):**
```
GET sqlite3:query(queries=["SELECT title, severity, status, date FROM alerts ORDER BY date DESC"]) AS report
```
`raw_parameters` режется на `queries=["SELECT title`, `severity`, `status`, `date FROM ... DESC"]`.
В `queries` попадает строка `["SELECT title` → при сверке типов:
`wrong parameter type for queries (put <class 'str'> (["SELECT title) need <class 'list'>)`.

Причина: split не учитывает кавычки и скобки `()[]{}`.

### P2 — split команд по `|` ломает SQL-конкатенацию и regex
```python
result = re.split(r'\|\s*', text, flags=re.DOTALL)
```
`|` ищется везде, в т.ч. внутри значений. SQLite-конкатенация `a || b`, побитовые операции,
символ `|` в регулярках/строках → ошибочное разбиение на «команды».

### P3 — разбор `AS` в APPLY-колонках сломан (приоритет альтернации)
```python
splitted = re.split(r'\s+as|AS|As|aS\s+', apply_column)
```
Из-за приоритета `|` это означает «`\s+as` ИЛИ `AS` ИЛИ `As` ИЛИ `aS\s+`». То есть split
срабатывает на голом `AS`/`As` где угодно (даже внутри слова, напр. `GAS`, `CLASS`),
и не требует пробелов вокруг. Колонки с такими подстроками разбираются неверно.

### P4 — нельзя параметризовать значение с запятыми/плейсхолдерами через `DEF`
`get_variable_type` для `DEF` требует **валидный JSON** уже на этапе парсинга. Поэтому
`DEF {"_from":%(start)i} AS f` не проходит (плейсхолдер ломает JSON), а инъекция работает
только в параметрах `GET`. В сочетании с P1 это означает: единственный обходной путь для
сложных значений (через `DEF` + `%(...)d`) работает лишь частично.

### P5 — удаление комментариев жадное и без DOTALL
```python
re.sub(r"/\*.*\*/", '', text)
```
`.*` жадная → между двумя отдельными комментариями в одной строке вырежется и код между ними;
без `DOTALL` многострочный комментарий не удаляется. (Команды разбиваются позже по `|`,
многострочность через `DOTALL` есть только там.)

### P6 — прочие хрупкости
- `^([^:]+):([^\(]+)\((.*)\)$` — имя source/функции не может содержать `:`/`(`; жадная `.*`
  до последней `)` (обычно ок, но без гарантий при `)` в значениях без кавычек).
- Значения параметров остаются строками и типизируются «угадыванием» позже: строка,
  похожая на число/список, будет приведена к нему принудительно.
- `DEF`/`GET` `AS`: имя-приёмник — `\S+` (имя данных не может содержать пробелы).
- Нет механизма экранирования кавычек/скобок внутри значений.

## 4. Предложения по исправлению

### Fix #1 (обязательный) — split по запятой только на верхнем уровне
Ввести разделитель, учитывающий кавычки и вложенность `()[]{}`, и применить его к параметрам
`GET` и к колонкам `APPLY`.

```python
def split_top_level(text, separator=','):
    """Split `text` by `separator` only at the top nesting level:
    ignores separators inside (), [], {} and inside '...'/\"...\" strings."""
    parts, buf = [], []
    depth = 0
    quote = None
    prev = ''
    for ch in text:
        if quote:
            buf.append(ch)
            if ch == quote and prev != '\\':
                quote = None
        elif ch in ('"', "'"):
            quote = ch
            buf.append(ch)
        elif ch in '([{':
            depth += 1
            buf.append(ch)
        elif ch in ')]}':
            depth = max(0, depth - 1)
            buf.append(ch)
        elif ch == separator and depth == 0:
            parts.append(''.join(buf))
            buf = []
        else:
            buf.append(ch)
        prev = ch
    parts.append(''.join(buf))
    return parts
```

Применение в `command_parser`:
```python
# было: parameters = re.split(r',\s*', command["raw_parameters"])
parameters = split_top_level(command["raw_parameters"], ',')
...
# было: apply_columns = re.split(r',\s*', command["apply"]["raw_columns"])
apply_columns = split_top_level(command["apply"]["raw_columns"], ',')
```
(каждый элемент далее `.strip(" ")` — как уже делается). После этого пример из P1
парсится как одно значение-строка `["SELECT ... DESC"]`, а сверка типов в
`commands_executor` приводит её к списку через `get_variable_type`.

### Fix #2 — split команд по `|` тоже на верхнем уровне
Использовать тот же `split_top_level(text, '|')` вместо `re.split(r'\|\s*', text)`, чтобы
`||` и `|` внутри строк/скобок не считались разделителями команд. (Дополнительно отфильтровать
пустые сегменты и сохранить поддержку многострочности.)

### Fix #3 — корректный разбор `AS` в APPLY
Заменить сломанный регэксп на регистронезависимый с границами слова и разбором по последнему
вхождению, например:
```python
m = re.match(r'^(?P<col>.+?)\s+[Aa][Ss]\s+(?P<alias>\S+)\s*$', apply_column.strip())
```
или `rsplit` по ` AS `/` as ` без захвата подстрок внутри слов.

### Fix #4 — параметризация сложных значений
Варианты (в порядке предпочтения):
1. Разрешить значениям `DEF` оставаться «сырой строкой», если в них есть `%(...)`,
   и типизировать уже **после** инъекции (перенести `get_variable_type` за `process_injections`).
2. Разрешить инъекцию переменных не только в `GET`, но и в `DEF`-значения (рекурсивно),
   с поздней типизацией.

### Fix #5 — комментарии
```python
text = re.sub(r"/\*.*?\*/", '', text, flags=re.DOTALL)   # нежадно + многострочно
```

### Долгосрочно — заменить регэкспы на нормальный лексер/грамматику
Текущий парсер — набор ad-hoc регэкспов, чувствительных к запятым, кавычкам, `|`, `AS`.
По мере роста DSL стоит перейти на явный токенайзер или PEG-грамматику (напр. `lark`):
- единые правила кавычек/экранирования/вложенности;
- внятные синтаксические ошибки с позицией;
- расширяемость (новые команды/операторы) без регрессий.
Это снимает классы багов P1–P3, P6 разом.

## 5. Приоритеты

1. **Fix #1** — разблокирует SQL/JSON-списки в параметрах (текущий блокер тестирования).
2. **Fix #2** — устраняет тихие поломки SQL с `||` и `|`.
3. **Fix #3** — корректный `APPLY`.
4. **Fix #4/#5** — удобство параметризации и комментарии.
5. **Лексер** — стратегически, когда DSL стабилизируется.

Минимальный безопасный шаг для текущего тестирования — внедрить `split_top_level` (Fix #1,
опц. #2). Изменение локально в `command_parser`, обратно совместимо для значений без запятых.
