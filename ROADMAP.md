## 1. Текущее состояние (на момент составления)

### Работает (проверено на стенде)
- Запуск приложения; **ленивая загрузка** зависимостей (старт с минимальным набором).
- **Единая сигнатура коннекторов** `(parameters, source_object, data_map, current_state)` — все 35 функций.
- **Хранилище секретов** (CRUD) в UI; шифрование Fernet.
- **Коннектор IRP TheHive 5.x** (`irp_thehive:get_alerts` по фильтру).
- **Парсер DSL**: корректный split по запятым/`|` с учётом кавычек и скобок; покрыт тестами (`tests/`).
- **Вывод результатов**: `PRINT` (markdown: текст / MD-таблица), `SHOW(table, table)` (aggrid с фильтрами/сортировкой, тултипы), `SHOW(table, matplotlib, {...})` (чёткий график, настраиваемый `dpi`).
- **Индикация выполнения**: спиннер в хэдере + панель шагов со статусами `validation/pending/running/done/error/rejected`, прерывание прогона на ошибке шага.
- Сворачиваемые блоки скрипта (с кнопкой Execute) и шагов.

### Осознанно принятые решения
- **SQL-инъекции в SQL-источниках и `app/db.py` — принятый риск** (доверенный ограниченный круг, роли, сетевой whitelist). См. `ARCHITECTURE.md §10`.

### Не реализовано / заглушки / техдолг
- DSL: **`CALC`** (парсится, не исполняется), **`SAVE`** (заглушка).
- Подсистемы: **AI-режим** (`draw_ai`, `app/llm.py` — каркас), **Settings/пользователи** (`draw_users` нет).
- **`universal_harvester:local_scenario`** ссылается на несуществующие `app.database.scenarios` / `app.engine.scenarios`.
- **`grafana`/`teleport`** — отключены в реестре; **API-эндпоинт** (`/api/...`) закомментирован.
- **Параллелизм** выключен (`multiprocessing.Pool` закомментирован); `max_threads`/`threads_limit` и `get_source_threads_pool` не задействованы.
- Таблицы **`executions`/`storage`** созданы, но не используются (нет истории запусков и сохранения данных).
- **Master key и зашифрованные конфиги захардкожены** в `front.py`.
- Тестами покрыт только парсер; на движок/коннекторы/UI тестов нет. Нет `pyproject`, `Dockerfile`, CI.
- Опечатки/мелочи: `crptgrphy.py`, `nane`, `dynamica_agg_dict`.

## 2. Принципы развития
- **Сначала надёжность ядра, потом расширение.** Не строить AI/MCP поверх неустойчивого исполнения.
- **Обратная совместимость DSL** — скрипты пользователей не должны ломаться; новые возможности аддитивны.
- **Каждая новая возможность — с тестом** (хотя бы парсер/движок офлайн без БД).
- **MCP/агент — стратегическая цель**: source как инструмент, который вызывает и человек, и агент.

## 3. Фазы развития (предложение)

### Фаза 1. Завершение DSL-петли и устойчивость ядра  ⟵ рекомендую начать здесь
Цель: скрипт пишется и полностью отрабатывает предсказуемо, ошибки видны.
- `SAVE` — сохранение результатов в таблицу `storage` (owner/execution/json) + чтение.
- `CALC` — базовые операции над переменными (числа/строки/даты), с тестами парсера и исполнения.
- Тесты движка `commands_executor` офлайн: граф зависимостей, `APPLY`, прерывание на ошибке, статусы.
- Мелкий техдолг: исправить опечатки имён (с осторожностью к обратной совместимости источников).
- **Критерий готовности**: типовой сценарий (несколько `GET` → SQL/pandas → `PRINT/SHOW/SAVE`) проходит, повторно открывается из `storage`, все ветки ошибок отражаются в панели шагов.

### Фаза 2. Выполнение и история
Цель: масштабируемое и наблюдаемое выполнение.
- **История запусков** (`executions`): запись id/owner/timestamp/status/json; вкладка просмотра прошлых прогонов и их результатов.
- **Параллелизм по стадиям**: включить пул (нитями, не процессами — из-за UI-refs в `current_state`), задействовать `threads_limit`; унифицировать `max_threads`/`threads_limit`.
- Под-прогресс длинных коннекторов (например, постраничная выгрузка TheHive) → в панель шагов.
- **Критерий готовности**: сценарий с независимыми `GET` выполняется параллельно; прошлый прогон открывается из истории.

### Фаза 3. API / MCP-слой  ⟵ стратегическая цель
Цель: source/script вызываются программно — это и есть «конструктор MCP».
- Оживить REST-эндпоинт: запуск сохранённого `script` по имени с параметрами, выдача результата (JSON/файл).
- Аутентификация для API (токен/ключ), авторизация по ролям, аудит вызовов.
- **MCP-сервер**: экспонировать каждый зарегистрированный source-`function` как MCP-tool со схемой параметров (из `required`), чтобы внешний агент мог их вызывать.
- **Критерий готовности**: внешний клиент/агент вызывает `script` и конкретный source через API/MCP с проверкой прав.

### Фаза 4. AI-режим (агент-сборщик сценариев)
Цель: из запроса на естественном языке — готовый DSL-скрипт.
- Довести `app/llm.py`: пайплайн «запрос → выбор источников/инструментов → сборка DSL → выполнение → ответ».
- Реестр источников как набор инструментов для function-calling; долговременная память/контекст (раздел «Knowledge base» уже намечен).
- **Критерий готовности**: по NL-запросу агент собирает корректный скрипт, исполняет его движком и возвращает результат с источниками.

### Фаза 5. Эксплуатация и качество
Идёт частично параллельно остальным фазам.
- **Конфигурация без хардкода**: master key и конфиги — через env/secret-manager/интерактивный ввод; убрать значения из `front.py`.
- `pyproject.toml`/lock, `Dockerfile`, healthcheck; CI (линт + `python -m unittest`).
- Наблюдаемость: структурные логи (уже JSON), метрики выполнения, тайминги шагов.
- Расширение тестов: движок, рендер-хелперы (вынести чистые функции из `interface.py` в модуль без `nicegui` для тестируемости).

## 4. Бэклог коннекторов (по мере потребности)
- Починить/перенести подсистему **сценариев** (`universal_harvester`).
- Вернуть **grafana**/**teleport** под актуальную сигнатуру и реестр.
- Новые источники по запросу (паттерн — `ADDING_SOURCES.md`); для TheHive — авто-пагинация и доп. сущности (cases, observables).

## 5. Сквозные направления
- **Безопасность**: в рамках принятой модели угроз; при расширении круга пользователей вернуться к параметризации `app/db.py` и валидации токенов (чеклист в `ARCHITECTURE.md`).
- **Документация**: поддерживать `ARCHITECTURE.md` актуальным; руководство пользователя по DSL (DEF/GET/APPLY/CALC/SAVE/PRINT/SHOW/NOTIFY) с примерами.

## 6. Открытые вопросы для приоритизации
1. Что важнее раньше — **завершить DSL (Фаза 1)** или сразу двигаться к **API/MCP (Фаза 3)**?
2. Нужна ли **история запусков** (Фаза 2) в ближайшей итерации или достаточно `SAVE`?
3. AI-режим (Фаза 4): локальные модели (ollama/llama) или внешний провайдер? Это влияет на дизайн `llm.py`.
4. Параллелизм: достаточно ли нитей (текущая модель `run.io_bound`), или планируются CPU-тяжёлые шаги (тогда нужен иной канал прогресса)?
